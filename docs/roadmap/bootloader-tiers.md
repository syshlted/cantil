# Roadmap: Bootloader configuration tiers (Kconfig choice)

**Bootloader configuration tiers (Kconfig choice).** Three build-time options, each with a different security/convenience tradeoff. Selected via a `choice CANTIL_BOOTLOADER` block in Kconfig; downstream MCUboot + UF2 config keys are forced from the choice so they can't drift.
  - **`CANTIL_BOOTLOADER_UF2_ONLY`** (development default) — current behaviour. Adafruit UF2 bootloader writes `.uf2` files directly to the app partition. No signature validation, no rollback, no A/B. `cantil dfu` (1200bps-touch) reboots into UF2. Fastest iteration; appropriate for dev builds.
  - **`CANTIL_BOOTLOADER_UF2_MCUBOOT`** (convenience + rollback) — UF2 stays as the bottom layer for recovery flashing; MCUboot is added inside the app partition with primary slot internal and secondary slot on QSPI. The build pipeline produces signed `.uf2` files whose target addresses point at the **secondary slot**. UF2 itself does no validation (it can't — it just writes bytes); the validity gate is MCUboot on the next reboot. Flow: user drops `.uf2` → UF2 writes to secondary slot → reboot → MCUboot verifies signature against its baked-in public key → swap on success, keep old primary on failure (with `LED_PATTERN_FW_INVALID` and tap-confirm choice per the entry below). Known weakness: a hand-crafted `.uf2` targeting primary-slot addresses bypasses MCUboot entirely. Mitigation options ranked: (1) fork TinyUF2 to refuse writes outside the secondary-slot range — best security, ongoing fork maintenance; (2) document the limitation — this tier is positioned as "drag-and-drop + accidental-bad-build protection," not adversarial. Default to (2); revisit (1) if the fork burden looks manageable.
  - **`CANTIL_BOOTLOADER_MCUBOOT`** (production / maximum security) — UF2 bootloader removed from flash entirely. MCUboot occupies the boot region. Firmware updates flow only through the authenticated `UPDATE_FIRMWARE` opcode over the Noise session (see the entry below). Recovery requires SWD (XIAO exposes pads only on tiny underside test points — document the procedure). `cantil dfu` shell command is compiled out. Combined with APPROTECT, this is the tier that holds up against a device-in-hand attacker.
  - **Build pipeline implications:** signed-image build is needed for both MCUboot tiers. UF2-only tier skips signing. The `.uf2` wrapper step in the MCUboot tier must target the secondary slot's base address (not the app partition base) — needs an explicit `--base-address` flag in the `uf2conv` invocation, plumbed from devicetree partition info. `scripts/build.sh` should pick the right pipeline based on the active Kconfig choice.
  - **Test plan:** smoke-test each tier on a real XIAO. UF2-only: drop unsigned `.uf2`, app runs. UF2+MCUboot: (a) drop signed-with-correct-key `.uf2` → swaps and runs new image; (b) drop signed-with-wrong-key `.uf2` → MCUboot rejects, old image keeps running, LED warns; (c) drop unsigned `.uf2` → same rejection path. MCUboot-only: confirm UF2 drive does not mount on double-tap-reset, confirm `UPDATE_FIRMWARE` over Noise completes a swap.
