# Roadmap: Zephyr userspace / privilege separation for the CA signing path

**Zephyr userspace / privilege separation for the CA signing path.** Zephyr's `CONFIG_USERSPACE` uses the nRF52840's ARMv7-M MPU to enforce per-thread memory isolation: a thread in its own `k_mem_domain` can only touch memory in that domain, and kernel APIs are reachable only through validated syscalls. The applicable shape for Cantil is **not** to userspace the whole firmware — USB device stack, BLE host, mbedtls/PSA, and the CC310 driver all assume supervisor mode and porting them is not realistic — but to isolate the **CA signing worker** as a user thread in a private domain. It would receive a TBS-cert + slot ID over a `k_msgq`, syscall into CC310 to sign, and return the signature. The decrypted private key buffer would live in a partition that USB/CBOR/BLE code physically cannot read, so a parse bug in the I/O path can't exfil the CA key. Costs: the ARMv7-M MPU has only **8 regions** (text/rodata/BSS/stack/key-buffer/CC310 MMIO/flash window fill it fast), each user thread needs both a user stack and a privileged stack with power-of-two MPU-aligned sizes (probably 4–8 KB extra RAM), and `CONFIG_USERSPACE` machinery adds ~5–15 KB flash. Not worth doing yet — the CA key is in RAM only for milliseconds per sign (decrypt → CC310 → zeroize), CC310 locked mode keeps key material in CC310 registers (stronger than MPU isolation, already on the design list), and APPROTECT closes the more likely physical-access path (SWD). Queue behind: APPROTECT in production builds, MCUboot signed images, authenticated FW update opcode. Revisit during production hardening.
