## Future crypto/library options

- **OpenSSL libcrypto build target** — not feasible on nRF52840 (3–5 MB code, no PSA/CC310 hardware-acceleration path), but worth keeping in the roadmap for future ports to larger SoCs where the flash/RAM budget allows it. Would unlock richer X.509 tooling and ecosystem familiarity at the cost of giving up the unified-mbedTLS-everywhere story. Apache 2.0 since OpenSSL 3.0, so AGPL-compatible. Selection would follow the same Kconfig backend pattern as the Noise crypto backend.

## Future commands (not yet designed)

- Secure device pairing (`PAIR_DEVICE`, USB only): two nRF devices establish a persistent authenticated pairing via a client-proxied connection; neither device can decrypt the other's Noise session, but the client shuttles ciphertext between them. Both devices store the pairing bond; future connections between paired devices reject any intermediary that can't present the correct pairing identity. Tap confirmation on both devices required.
- Secure slot transfer (`TRANSFER_SLOT`): move a key slot from one paired device to another to avoid single-point-of-failure; receiving device must tap its unlock sequence to accept; slot placed in next free slot on receiver; fails if receiver has no free slots. Requires completed pairing.
- **Import a CA** (`IMPORT_CA` / `IMPORT_KEY` family): bring an externally-managed CA onto the device.
  - Minimum: private key import. Device must verify the key can actually sign (test-sign + verify round-trip on a synthetic TBS) before committing to a slot — otherwise reject. Client library mirrors this precondition before sending.
  - Optional bundle: signed CA cert (if one exists) + list of previously-issued certs with their serial index, so the cert store can be reconstructed.
  - Possible helper: importer that reads an OpenSSL-managed CA directory (`private/`, `certs/`, `index.txt`, `serial`) and feeds it to the device.
  - Open questions: protected-slot handling on import target, key-type negotiation (P-256 vs P-384 vs RSA), wire format (PKCS#8? raw scalar? encrypted?), and whether CSR-style proof-of-possession is enough vs. round-trip sign-then-verify.
- **Secure pairing between clients and devices**: an additional bond layered on top of the existing Noise_XX static-key auth.
  - Today: a device authenticates any client that presents a known static pubkey. With pairing, a device only accepts clients (or peer devices) that completed a pairing handshake — even possession of a valid static key is not sufficient without the pairing record.
  - Enables fleet deployments: many devices share a signing CA, but only explicitly paired client/device pairs can interoperate.
  - Also the foundation for device-to-device pairing (precondition for `TRANSFER_SLOT` and future "secure sync" between devices).
  - Open: pairing-record storage (LittleFS slot vs. dedicated partition), revocation/unpair flow, and whether tap confirmation is required on each pairing.
