# Cantil — Marketing Positioning

Scratchpad for taglines, audience framing, and pitch copy. Nothing here is final; this is where positioning gets argued out before it lands in `README.md` or anywhere external.

---

## What Cantil is

Cantil is a portable firmware project that turns a small MCU into a self-contained Certificate Authority. Private keys are generated and stored in hardware, never leave the device, and all host communication is end-to-end encrypted with Noise_XX over USB CDC/ACM or BLE NUS.

**The project is the product.** A specific board (XIAO BLE Sense, an nRF52840DK, etc.) is just the current carrier. The codebase is structured so new SoCs can be added via Kconfig + devicetree overlays without touching the core CA, session, or protocol layers.

**Current platform support:** any nRF52840 board. Reference target: Seeed Studio XIAO BLE Sense. Provided overlays: `xiao_ble`, `nrf52840dk/nrf52840`, `nrf52840dongle/nrf52840`, `adafruit_feather_nrf52840`, `generic_nrf52840`.

**Designed to extend.** New platforms slot in behind the existing Kconfig feature flags (`CANTIL_INPUT_*`, `CANTIL_STORAGE_*`, `CANTIL_TRANSPORT_*`). The CA, gesture state machine, Noise session, and CBOR protocol are platform-agnostic.

## Who it's for

**Audience 1 — the PKI owner.** Someone who wants to run real root and intermediate CAs offline, on hardware they physically control, without standing up an air-gapped laptop or paying for an HSM. Personal target for the project author: 4 roots + 16 intermediates = 20 CAs, fully managed on one device.

**Audience 2 — the embedded developer.** Someone building a constrained device (IoT product, sensor, appliance, hobby project) that needs certificates for secure pairing, mutual auth, or signing — but doesn't want to ship a full PKI implementation inside their own firmware. They link `libcantil`, point it at a Cantil device over USB or BLE, and offload every CA operation. The end-to-end Noise session means the transport can be untrusted: a sniffer sees ciphertext, MITM is structurally impossible.

Both audiences use the same firmware and the same client library. The difference is just who's holding the Cantil device — a human, or another piece of embedded code.

## Tagline

**Chosen headline:**

> **Tiny PKI. Big potential.**

Confident, product-marketing voice. Makes a virtue of the constraint — "tiny" carries double duty (tiny device, tiny attack surface). "Big potential" stays open-ended enough to cover both the PKI-owner and embedded-developer audiences without overclaiming any specific feature. Used as the README banner subtitle under the `# Cantil` heading.

**Subtitle / description lead-in** (keeps the playful dev-to-dev voice in the body):

> The embedded project you didn't realize you needed.

Woven into the README sub-line as a lead-in clause to the substantive pitch. Also reusable as the GitHub repo short description.

## Pitch — one paragraph

> Cantil is a portable firmware project that turns a small MCU into a self-contained Certificate Authority and Hardware Security Module. Generate and manage real root and intermediate CAs on a device you physically control — no air-gapped laptop, no cloud service, no subscription. Sign CSRs, issue certs, manage CRLs, and keep private keys in hardware that never reveals them. The companion client library, `libcantil`, makes the same CA available to other embedded projects: link it into your firmware, talk to a Cantil device over USB or BLE, and get secure pairing and mutual authentication without implementing PKI yourself. All host communication runs end-to-end encrypted via Noise_XX, so the physical transport is allowed to be untrusted.

---

## Pitch — three bullets

- **Real PKI, fully offline.** Root and intermediate CAs live on the device. Private keys are generated by a hardware TRNG, stored encrypted on the device's flash, and used inside the crypto accelerator — they never appear in host memory.
- **An embeddable CA for your other projects.** `libcantil` lets a constrained device call out to a Cantil instance for certificate operations. Your firmware stays small; your PKI stays consistent.
- **Untrusted transports welcome.** Every byte between client and device is Noise_XX-encrypted with mutual static-key authentication. USB sniffing, BLE eavesdropping, and MITM all see ciphertext only.
